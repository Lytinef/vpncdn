# Архитектура

## Общая схема

```
┌─────────────┐     HTTPS API      ┌──────────────┐     SQL      ┌────────────┐
│  Flutter    │ ─────────────────▶ │   Backend    │ ───────────▶ │ PostgreSQL │
│  клиент     │                    │   (NestJS)   │              └────────────┘
│ (Android)   │ ◀───────────────── │              │ ───────────▶ ┌────────────┐
└─────┬───────┘   токены/конфиг    │              │   очереди    │   Redis    │
      │                            └──────┬───────┘   (BullMQ)   └────────────┘
      │ VLESS+WS+TLS                      │  управление узлами (HTTP агент)
      ▼                                   ▼
┌─────────────┐                    ┌──────────────┐
│   NGENIX    │ ──────────────────▶│  Origin VPS  │
│   (CDN)     │   WS+TLS к origin  │  Xray + агент│ ──▶ Интернет
└─────────────┘                    └──────────────┘

         ┌──────────────┐   admin API
         │  Веб-админка │ ───────────▶ Backend
         │ (React+Vite) │
         └──────────────┘
```

## Компоненты

- **Backend (NestJS)** — единая точка истины: пользователи, подписки, платежи,
  устройства, узлы, список обхода. Модули:
  `auth` (Telegram+JWT), `subscriptions`, `payments` (YooKassa+рекуррент),
  `devices`, `nodes`, `xray` (генерация VLESS-конфигов и вызов агентов),
  `bypass`, `account`, `admin`.
- **PostgreSQL** — хранение (TypeORM + миграции).
- **Redis + BullMQ** — очередь автосписаний, периодические задачи (cron).
- **Flutter-клиент** — UI + нативное VPN-ядро (Android `VpnService` + Xray + tun2socks).
- **NGENIX (CDN)** — прослойка: терминирует TLS клиента и проксирует WS на origin.
- **Xray-узел** — origin-сервер с Xray (VLESS+WS+TLS) и агентом управления клиентами.
- **Админка (React)** — управление с любого устройства/IP.

## Поток подключения VPN

1. Клиент логинится (Telegram) → получает JWT.
2. При первом подключении регистрирует устройство (`POST /devices`) — backend
   проверяет лимит тарифа, выбирает наименее загруженный узел, генерирует VLESS UUID,
   через агент добавляет клиента на Xray-узел.
3. Клиент запрашивает `GET /devices/:id/connection` — получает параметры VLESS+WS+TLS
   (адрес = домен NGENIX).
4. Клиент берёт список обхода (`GET /bypass`), строит конфиг Xray, поднимает TUN,
   запускает Xray и tun2socks.
5. Трафик: TUN → tun2socks → SOCKS(Xray) → VLESS+WS+TLS → NGENIX → origin → интернет.
   Домены/приложения обхода идут мимо туннеля (правила Xray + per-app в VpnService).

## Безопасность

- Клиентские токены: короткий access + ротируемый refresh (сессии в БД, хэш токена).
- Админка: отдельный JWT, bcrypt-пароли.
- YooKassa webhook: статус перепроверяется запросом к API (тело не доверяется).
- Telegram-вход: проверка HMAC-подписи Login Widget + срок действия.
- Агент узла: Bearer-секрет, порт закрыт фаерволом для backend.
